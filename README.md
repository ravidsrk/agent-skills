# рҹӨ– Multi-agent orchestration (Orca)

### Architecture (all multi-agent skills)

```
Orca runtime + orchestration skill   вҶҗ HARD BASE (from Orca CLI; not this repo)
        в”Ӯ
        в”ңв”Җв”Җ clean-sweep / spec-to-ship     (this pack вҖ” product/audit peers)
        в””в”Җв”Җ matt-ship / wayfinder-fleet / вҖҰ (this pack вҖ” MattГ—Orca peers)
                в”Ӯ
                в””в”Җв”Җ worker playbooks: mattpocock/skills (/implement, /tdd, вҖҰ)
```

**Nothing multi-agent here runs without Orca orchestration.** Matt skills are *what workers run*; Orca is *how the coordinator dispatches, waits, and gates*. Skills in this pack do **not** depend on each other at runtime (e.g. clean-sweep is not built on matt-ship).


All require **Orca + `orchestration` skill (Orca CLI)**. MattГ—Orca skills also need [mattpocock/skills](https://github.com/mattpocock/skills). Shared helpers: [`scripts/orca-coord/`](scripts/orca-coord/).

## Install matrix

| Track | Skills | Extra runtime |

| **A вҖ” Capability** (any agent harness) | `cloudflare-dns`, `namecheap-dns`, `fly-to-aws-migration`, `deep-research`, `terminal-poster` | Env keys only (see below) |
| **B вҖ” Orca multi-agent** | `clean-sweep`, `spec-to-ship` | Orca + `orchestration` (Orca CLI). Peers; neither depends on the other. |
| **C вҖ” Matt Г— Orca** | `matt-ship`, `wayfinder-fleet`, `triage-to-fleet`, `ready-agent-drain`, `review-matrix`, `adversarial-ticket`, `diagnose-swarm`, `architecture-sprint`, `design-it-thrice`, `research-then-grill`, `model-jury`, `content-wayfinder` | Orca + `orchestration` + **mattpocock/skills** for worker playbooks. |

рҹҹЎ **Name collision:** this repo's `deep-research` is the **monid 8-source** orchestrator. Other skill packs (e.g. makerskills) may ship a different skill with the same name. Symlinking this repo's copy will replace the other under `~/.claude/skills/deep-research`. Keep makerskills under `~/.agents/skills/` if you need both.

# Set the env vars

Each skill needs different secrets. Set them in your shell вҖ” never paste into prompts.

```bash
# cloudflare-dns
export CLOUDFLARE_API_KEY=cfat_...
export CLOUDFLARE_GLOBAL_API_KEY=...    # only for new-zone creation
export CLOUDFLARE_EMAIL=you@example.com

# namecheap-dns
export NAMECHEAP_API_KEY=...
export NAMECHEAP_API_USER=your-account

# fly-to-aws-migration
export AWS_PROFILE=migration
export FLY_API_TOKEN=...
# (also uses CLOUDFLARE_API_KEY for the DNS cutover phase)

# deep-research
export MONID_API_KEY=...

# terminal-poster
export OPENROUTER_API_KEY=...

# clean-sweep / spec-to-ship вҖ” no API keys; need Orca runtime + orchestration skill (from Orca CLI)
# orca status --json   # must show running; enable orchestration in Experimental settings
```

рҹ”ҙ **All skills read env vars at runtime вҖ” they're never written to disk.** Don't commit `.env` files.

**Key design choices:**

- рҹҹў **Process, not prose.** Skills are workflows agents follow, not reference docs they read.
- рҹҹў **Spec-compliant.** Every skill validates against the [agentskills.io](https://agentskills.io/specification) format.
- рҹҹў **Battle-tested.** These are the actual tools used in production, not theoretical patterns.
- рҹҹў **Progressive disclosure.** `SKILL.md` is the entry point. References load on demand to keep token usage low.
- рҹҹў **Capability-first, not SDLC-first.** Unlike skill packs that map to DEFINE вҶ’ PLAN вҶ’ BUILD вҶ’ SHIP, these are discrete tools. Pick what you need.

# Why this repo exists

Most agent skill packs are shaped like a software lifecycle вҖ” DEFINE вҶ’ PLAN вҶ’ BUILD вҶ’ SHIP. That's useful for skills that encode process discipline (writing specs, doing TDD, reviewing code).

This repo is shaped differently. These are **capability skills** вҖ” tools the agent reaches for when the task fits. You don't migrate to AWS as part of a SDLC phase; you migrate when you migrate. You don't run deep research because the workflow says so; you run it when you need evidence.

The two styles are complementary. For SDLC discipline, [addyosmani/agent-skills](https://github.com/addyosmani/agent-skills) is excellent. For real-world infrastructure and content tooling, this repo is what I reach for.

The shared bet: **portable, public, spec-compliant** skills make agents better.
